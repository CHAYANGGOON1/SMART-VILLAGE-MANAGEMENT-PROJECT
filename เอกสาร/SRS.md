# เอกสารข้อกำหนดความต้องการซอฟต์แวร์ (Software Requirements Specification - SRS)

## สำหรับระบบบริหารจัดการหมู่บ้านอัจฉริยะ (Smart Village Management System)

---

**เวอร์ชัน:** 1.0  
**วันที่:** 15 กรกฎาคม 2568  
**ผู้จัดทำ:** คณะผู้จัดทำโครงการ

---

### 1. บทนำ

#### 1.1 วัตถุประสงค์
เอกสารข้อกำหนดความต้องการซอฟต์แวร์ (SRS) ฉบับนี้มีวัตถุประสงค์เพื่อระบุข้อกำหนดด้านฟังก์ชันการทำงานและข้อกำหนดที่ไม่ใช่ฟังก์ชันการทำงานสำหรับ **ระบบบริหารจัดการหมู่บ้านอัจฉริยะ (Smart Village Management System)** โดยละเอียด ระบบนี้มีเป้าหมายที่จะเป็นแพลตฟอร์มดิจิทัลที่ครอบคลุมสำหรับผู้อยู่อาศัยและผู้ดูแลระบบของหมู่บ้าน เพื่อจัดการกิจกรรมต่าง ๆ ที่เกี่ยวข้องกับชุมชน การสื่อสาร และบริการต่าง ๆ ได้อย่างมีประสิทธิภาพ

#### 1.2 ขอบเขตของระบบ
**ระบบบริหารจัดการหมู่บ้านอัจฉริยะ** จะครอบคลุมฟังก์ชันการทำงานสำหรับการจัดการผู้ใช้งาน, การประกาศข่าวสาร, การแจ้งซ่อม, การจองพื้นที่ส่วนกลาง, การจัดการบิลและการชำระเงิน, การแบ่งปันเอกสาร, การจัดการความปลอดภัย (การลงทะเบียนผู้มาเยือน, การรายงานเหตุการณ์), การสำรวจความคิดเห็น (โหวต), และการสนทนาแบบเรียลไทม์ ระบบจะประกอบด้วยส่วน **Backend** ที่พัฒนาด้วย Flask และส่วน **Frontend** ที่เป็นเว็บอินเทอร์เฟซ

#### 1.3 คำจำกัดความ, คำย่อ และอักษรย่อ

- **SRS:** Software Requirements Specification (เอกสารข้อกำหนดความต้องการซอฟต์แวร์)  
- **API:** Application Programming Interface (ส่วนเชื่อมต่อโปรแกรมประยุกต์)  
- **CORS:** Cross-Origin Resource Sharing (การแบ่งปันทรัพยากรข้ามต้นทาง)  
- **UUID:** Universally Unique Identifier (ตัวระบุเฉพาะสากล)  
- **SQLAlchemy:** Python SQL Toolkit and Object Relational Mapper (ชุดเครื่องมือ SQL และตัวจัดการความสัมพันธ์เชิงวัตถุสำหรับ Python)  
- **SocketIO:** ไลบรารี JavaScript สำหรับแอปพลิเคชันเว็บแบบเรียลไทม์  
- **ผู้อยู่อาศัย (Resident):** ผู้ใช้งานที่ลงทะเบียนและอาศัยอยู่ในหมู่บ้าน  
- **ผู้ดูแลระบบ (Admin):** ผู้ใช้งานที่ลงทะเบียนและมีสิทธิ์ในการดูแลจัดการระบบ  
- **CRUD:** Create, Read, Update, Delete (สร้าง, อ่าน, อัปเดต, ลบ)  

#### 1.4 เอกสารอ้างอิง

- `app.py` (ซอร์สโค้ด Backend)  
- `index.html` (ซอร์สโค้ด Frontend)  
- `script.js` (JavaScript ของ Frontend)  
- `style.css` (CSS ของ Frontend)  
- `smart_village.db` (ฐานข้อมูล SQLite)

### 2. ภาพรวมของระบบ

#### 2.1 มุมมองผลิตภัณฑ์
**ระบบบริหารจัดการหมู่บ้านอัจฉริยะ** เป็นแอปพลิเคชันแบบ Standalone ที่ออกแบบมาเพื่อปรับปรุงการดำเนินงานภายในหมู่บ้านจัดสรร ระบบจะทำงานร่วมกับฐานข้อมูล **SQLite** สำหรับการจัดเก็บข้อมูล และใช้ **WebSockets (SocketIO)** สำหรับการสื่อสารแบบเรียลไทม์

#### 2.2 ฟังก์ชันการทำงานของผลิตภัณฑ์
ระบบมีฟังก์ชันการทำงานหลักดังต่อไปนี้:

- **การจัดการผู้ใช้งาน:** การลงทะเบียน, การเข้าสู่ระบบ, การจัดการโปรไฟล์, และการควบคุมบทบาทและสถานะของผู้ใช้งานโดยผู้ดูแลระบบ  
- **การประกาศข่าวสาร:** การสร้าง, การดู, และการจัดการประกาศของหมู่บ้านโดยผู้ดูแลระบบ  
- **การแจ้งซ่อม:** ผู้อยู่อาศัยสามารถส่งคำขอแจ้งซ่อม, ติดตามสถานะ, และผู้ดูแลระบบสามารถจัดการคำขอเหล่านี้ได้  
- **การจองพื้นที่ส่วนกลาง:** ผู้อยู่อาศัยสามารถจองพื้นที่ส่วนกลาง และผู้ดูแลระบบสามารถอนุมัติ/ปฏิเสธการจองได้  
- **การจัดการบิลและการชำระเงิน:** ผู้ดูแลระบบสามารถออกบิลค่าใช้จ่ายให้กับผู้อยู่อาศัย (รายบุคคลหรือทั้งหมด) และผู้อยู่อาศัยสามารถดูบิล, ส่งหลักฐานการชำระเงิน, และติดตามสถานะการชำระเงินได้ ผู้ดูแลระบบสามารถตรวจสอบและอนุมัติการชำระเงิน  
- **การแบ่งปันเอกสาร:** ผู้ใช้งานสามารถอัปโหลดและจัดการเอกสารส่วนตัวได้ และผู้ดูแลระบบสามารถจัดการเอกสารสาธารณะได้  
- **การจัดการความปลอดภัย:** ผู้อยู่อาศัยสามารถลงทะเบียนผู้มาเยือนและรายงานเหตุการณ์ด้านความปลอดภัยได้ ผู้ดูแลระบบสามารถดูและจัดการบันทึกเหล่านี้ได้  
- **การสำรวจความคิดเห็น (โหวต):** ผู้ดูแลระบบสามารถสร้างโพลสำรวจ และผู้อยู่อาศัยสามารถเข้าร่วมการโหวตได้  
- **การสนทนาแบบเรียลไทม์:** ผู้ใช้งานสามารถสื่อสารแบบเรียลไทม์ผ่านห้องสนทนาทั่วไปและห้องสนทนาเฉพาะกลุ่ม  
- **ปฏิทินกิจกรรม:** การจัดการและการดูเหตุการณ์สำคัญของหมู่บ้าน  

#### 2.3 คุณลักษณะของผู้ใช้งาน

- **ผู้อยู่อาศัย:**  
  - ระดับความรู้ด้านเทคนิค: มีความรู้พื้นฐานในการใช้งานคอมพิวเตอร์/สมาร์ทโฟน  
  - ความต้องการ: เข้าถึงข้อมูลหมู่บ้านได้ง่าย, สามารถขอรับบริการ, ติดตามค่าใช้จ่ายส่วนตัว, และสื่อสารกับฝ่ายบริหาร/เพื่อนบ้านได้  

- **ผู้ดูแลระบบ:**  
  - ระดับความรู้ด้านเทคนิค: มีความรู้ปานกลางในการใช้งานคอมพิวเตอร์และการจัดการระบบ  
  - ความต้องการ: เครื่องมือในการจัดการผู้ใช้งาน, ออกประกาศ, ดูแลคำขอบริการ, จัดการการเงิน, และดูแลบันทึกความปลอดภัย  

#### 2.4 ข้อจำกัดทั่วไป

- ฐานข้อมูล: SQLite (สามารถขยายไปใช้ PostgreSQL ได้ในอนาคต)  
- Backend: Python Flask  
- Frontend: HTML, CSS, JavaScript (การเรนเดอร์ฝั่งไคลเอนต์)  
- การสื่อสารแบบเรียลไทม์: SocketIO  
- ความปลอดภัย: การเข้ารหัสรหัสผ่าน (Werkzeug.security), การยืนยันตัวตนพื้นฐาน (ชื่อผู้ใช้/รหัสผ่าน)  
- การจัดเก็บไฟล์: ไฟล์ที่อัปโหลด (รูปภาพ, เอกสาร) จะถูกจัดเก็บในระบบไฟล์ของเซิร์ฟเวอร์  

#### 2.5 ข้อสมมติฐานและข้อพึ่งพิง

- ผู้ใช้งานสามารถเข้าถึงอินเทอร์เน็ตเพื่อใช้งานเว็บแอปพลิเคชัน  
- เซิร์ฟเวอร์ Backend ทำงานและสามารถเข้าถึงได้  
- ผู้ใช้งาน Admin และ Resident เริ่มต้นจะถูกสร้างขึ้นเมื่อระบบทำงานครั้งแรก หากฐานข้อมูลว่างเปล่า  
- การอัปโหลดไฟล์จะถูกจัดการโดย Backend และจัดเก็บในโฟลเดอร์ `static/uploads` ที่กำหนดไว้

### 3. ข้อกำหนดเฉพาะ

#### 3.1 ข้อกำหนดด้านฟังก์ชันการทำงาน (Functional Requirements)

##### 3.1.1 การจัดการผู้ใช้งาน

- **FR1.1.1 การลงทะเบียนผู้ใช้งาน:**  
  ผู้ใช้งานใหม่สามารถลงทะเบียนได้โดยระบุชื่อ, ชื่อผู้ใช้, รหัสผ่าน, เบอร์โทรศัพท์, และที่อยู่  
  **ข้อมูลนำเข้า:** ชื่อ, ชื่อผู้ใช้, รหัสผ่าน, เบอร์โทรศัพท์, ที่อยู่  
  **ข้อมูลส่งออก:** ข้อความแจ้งความสำเร็จ/ล้มเหลวในการลงทะเบียน สถานะบัญชีเริ่มต้นเป็น 'pending'  
  **เงื่อนไขเบื้องต้น:** ชื่อผู้ใช้ต้องไม่ซ้ำกัน  
  **เงื่อนไขภายหลัง:** มีการสร้างบันทึกผู้ใช้งานใหม่ในฐานข้อมูล  

- **FR1.1.2 การเข้าสู่ระบบผู้ใช้งาน:**  
  ผู้ใช้งานที่ลงทะเบียนสามารถเข้าสู่ระบบด้วยชื่อผู้ใช้และรหัสผ่าน  
  **ข้อมูลนำเข้า:** ชื่อผู้ใช้, รหัสผ่าน  
  **ข้อมูลส่งออก:** โทเค็นการยืนยันตัวตน/เซสชัน, บทบาทผู้ใช้งาน, และสถานะ  
  **เงื่อนไขเบื้องต้น:** บัญชีผู้ใช้งานต้องมีอยู่และมีสถานะ 'approved'  
  **เงื่อนไขภายหลัง:** มีการสร้างเซสชันผู้ใช้งาน  

- **FR1.1.3 การจัดการโปรไฟล์ผู้ใช้งาน (ผู้อยู่อาศัย):**  
  ผู้อยู่อาศัยสามารถดูและอัปเดตชื่อ, เบอร์โทรศัพท์, และที่อยู่ของตนเองได้ พวกเขายังสามารถเปลี่ยนรหัสผ่านได้  
  **ข้อมูลนำเข้า:** ชื่อ, เบอร์โทรศัพท์, ที่อยู่ (ที่อัปเดต), รหัสผ่านปัจจุบัน (สำหรับการเปลี่ยนรหัสผ่าน), รหัสผ่านใหม่  
  **ข้อมูลส่งออก:** ข้อความแจ้งความสำเร็จ/ล้มเหลวในการอัปเดตโปรไฟล์  

- **FR1.1.4 การจัดการผู้ใช้งาน (ผู้ดูแลระบบ):**  
  ผู้ดูแลระบบสามารถดูผู้ใช้งานทั้งหมด, อนุมัติ/ระงับ/ปฏิเสธบัญชีผู้ใช้งาน, และแก้ไขรายละเอียดผู้ใช้งาน (ชื่อ, ชื่อผู้ใช้, เบอร์โทรศัพท์, ที่อยู่, บทบาท, สถานะ)  
  **ข้อมูลนำเข้า:** รหัสผู้ใช้งาน, ข้อมูลผู้ใช้งานที่อัปเดต (ชื่อ, ชื่อผู้ใช้, เบอร์โทรศัพท์, ที่อยู่, บทบาท, สถานะ)  
  **ข้อมูลส่งออก:** ข้อความแจ้งความสำเร็จ/ล้มเหลวในการอัปเดตผู้ใช้งาน  
  **เงื่อนไขเบื้องต้น:** ต้องมีสิทธิ์ผู้ดูแลระบบ  

##### 3.1.2 การจัดการประกาศข่าวสาร

- **FR1.2.1 การดูประกาศ (ผู้ใช้งานทุกคน):**  
  ผู้ใช้งานทุกคนสามารถดูรายการประกาศที่เผยแพร่แล้ว โดยเรียงตามวันที่  
  **ข้อมูลส่งออก:** รายการประกาศพร้อมหัวข้อ, เนื้อหา, วันที่เผยแพร่, ผู้เขียน, และแท็ก  

- **FR1.2.2 การสร้างประกาศ (ผู้ดูแลระบบ):**  
  ผู้ดูแลระบบสามารถสร้างประกาศใหม่พร้อมหัวข้อ, เนื้อหา, วันที่เผยแพร่, และแท็กเสริม  
  **ข้อมูลนำเข้า:** หัวข้อ, เนื้อหา, วันที่เผยแพร่, รหัสผู้เขียน, แท็ก (เช่น 'สำคัญ', 'กิจกรรม', 'แจ้งเตือน')  
  **ข้อมูลส่งออก:** ข้อความแจ้งความสำเร็จ/ล้มเหลวในการสร้างประกาศ **(พร้อมการแจ้งเตือนแบบเรียลไทม์)**  

- **FR1.2.3 การแก้ไขประกาศ (ผู้ดูแลระบบ):**  
  ผู้ดูแลระบบสามารถแก้ไขประกาศที่มีอยู่ได้  
  **ข้อมูลนำเข้า:** รหัสประกาศ, หัวข้อที่อัปเดต, เนื้อหา, วันที่เผยแพร่, แท็ก  
  **ข้อมูลส่งออก:** ข้อความแจ้งความสำเร็จ/ล้มเหลวในการอัปเดตประกาศ **(พร้อมการแจ้งเตือนแบบเรียลไทม์)**  

- **FR1.2.4 การลบประกาศ (ผู้ดูแลระบบ):**  
  ผู้ดูแลระบบสามารถลบประกาศได้  
  **ข้อมูลนำเข้า:** รหัสประกาศ  
  **ข้อมูลส่งออก:** ข้อความแจ้งความสำเร็จ/ล้มเหลวในการลบประกาศ **(พร้อมการแจ้งเตือนแบบเรียลไทม์)**  

##### 3.1.3 การจัดการคำขอแจ้งซ่อม

- **FR1.3.1 การส่งคำขอแจ้งซ่อม (ผู้อยู่อาศัย):**  
  ผู้อยู่อาศัยสามารถส่งคำขอแจ้งซ่อมใหม่พร้อมหัวข้อ, หมวดหมู่, รายละเอียด, และรูปภาพประกอบ (ไม่บังคับ)  
  **ข้อมูลนำเข้า:** รหัสผู้ใช้งาน, หัวข้อ, หมวดหมู่ (เช่น 'ไฟฟ้า', 'น้ำประปา'), รายละเอียด, ไฟล์รูปภาพ  
  **ข้อมูลส่งออก:** ข้อความแจ้งความสำเร็จ/ล้มเหลวในการส่งคำขอ **(พร้อมการแจ้งเตือนแบบเรียลไทม์ไปยังผู้ดูแลระบบ)**  

- **FR1.3.2 การดูคำขอแจ้งซ่อม (ผู้อยู่อาศัย):**  
  ผู้อยู่อาศัยสามารถดูรายการคำขอแจ้งซ่อมที่ตนเองส่งไปและสถานะปัจจุบัน  
  **ข้อมูลส่งออก:** รายการคำขอแจ้งซ่อมพร้อมรหัส, หัวข้อ, หมวดหมู่, วันที่ส่ง, และสถานะ  

- **FR1.3.3 การจัดการคำขอแจ้งซ่อม (ผู้ดูแลระบบ):**  
  ผู้ดูแลระบบสามารถดูคำขอแจ้งซ่อมทั้งหมด, อัปเดตสถานะ (รอรับเรื่อง, กำลังดำเนินการ, เสร็จสิ้น, ถูกปฏิเสธ), และดูรายละเอียดรวมถึงรูปภาพที่แนบมา  
  **ข้อมูลนำเข้า:** รหัสคำขอ, สถานะใหม่  
  **ข้อมูลส่งออก:** ข้อความแจ้งความสำเร็จ/ล้มเหลวในการอัปเดตคำขอ **(พร้อมการแจ้งเตือนแบบเรียลไทม์ไปยังผู้อยู่อาศัยที่ร้องขอ)**  

##### 3.1.4 การจัดการการจองพื้นที่ส่วนกลาง

- **FR1.4.1 การส่งคำขอจอง (ผู้อยู่อาศัย):**  
  ผู้อยู่อาศัยสามารถขอจองพื้นที่ส่วนกลาง (เช่น 'สนามกีฬา', 'คลับเฮ้าส์') สำหรับวันที่และช่วงเวลาที่ระบุ พร้อมวัตถุประสงค์และจำนวนผู้เข้าร่วม  
  **ข้อมูลนำเข้า:** รหัสผู้ใช้งาน, สถานที่, วันที่, เวลาเริ่มต้น, เวลาสิ้นสุด, วัตถุประสงค์, จำนวนผู้เข้าร่วม  
  **ข้อมูลส่งออก:** ข้อความแจ้งความสำเร็จ/ล้มเหลวในการส่งคำขอจอง **(พร้อมการแจ้งเตือนแบบเรียลไทม์ไปยังผู้ดูแลระบบ)**  

- **FR1.4.2 การดูคำขอจอง (ผู้อยู่อาศัย):**  
  ผู้อยู่อาศัยสามารถดูรายการคำขอจองของตนเองและสถานะปัจจุบัน  
  **ข้อมูลส่งออก:** รายการคำขอจองพร้อมรหัส, สถานที่, วันที่, เวลา, และสถานะ  

- **FR1.4.3 การจัดการคำขอจอง (ผู้ดูแลระบบ):**  
  ผู้ดูแลระบบสามารถดูคำขอจองทั้งหมดและอนุมัติ/ปฏิเสธได้  
  **ข้อมูลนำเข้า:** รหัสการจอง, สถานะใหม่  
  **ข้อมูลส่งออก:** ข้อความแจ้งความสำเร็จ/ล้มเหลวในการอัปเดตการจอง
